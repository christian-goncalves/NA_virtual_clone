````markdown
```text
Site oficial NA
↓

↓
Parser
````

## TAREFAS DESTA ETAPA

Implementar apenas os itens abaixo:

### 1. Service de sincronização

Criar o service:

`app/Services/NaVirtualMeetingSyncService.php`

Responsabilidades esperadas:

* buscar o HTML da página oficial com `Http::timeout(...)`
* usar `Symfony DomCrawler` para fazer o parsing
* extrair os dados das reuniões
* normalizar os dados
* salvar ou atualizar no banco usando o model `VirtualMeeting`
* marcar `synced_at`
* gerar uma chave estável para persistência idempotente
* registrar `source_url`
* registrar `source_hash`
* desativar reuniões que não forem encontradas na rodada atual, se isso for viável com segurança

O guia do projeto define esse service como responsável por baixar HTML, parsear reuniões, transformar em estrutura de dados e persistir localmente. 

---

### 2. Parser com DomCrawler

Usar:

* `Symfony DomCrawler`
* `Symfony CssSelector`

A lógica de parsing deve:

* localizar os blocos/itens de reunião na página oficial
* extrair, quando disponíveis:

  * nome
  * horário
  * link da reunião
  * plataforma
  * meeting id
  * senha
  * labels de formato
  * tipo
  * interesses
  * localidade
* tratar ausência de campos sem quebrar a sincronização
* aplicar fallback seguro quando algum campo não existir

A estratégia prática prevista no projeto é usar `Http::timeout(...)`, `Crawler($html)` e extratores específicos para os campos relevantes. 

---

### 3. Command manual

Criar o command:

`na:sync-virtual-meetings`

Responsabilidades:

* executar o `NaVirtualMeetingSyncService`
* permitir rodar a sincronização manualmente pelo terminal
* imprimir um resumo objetivo da execução, por exemplo:

  * total encontrado
  * total criado
  * total atualizado
  * total inativado
  * erros encontrados

## REGRAS IMPORTANTES

1. Implementar apenas o escopo da Etapa 2.

2. Não criar controller nesta etapa.

3. Não criar Blade nesta etapa.

4. Não criar scheduler ou job ainda.

5. Não criar API.

6. A coleta deve consumir a fonte oficial e alimentar a base local, sem depender da página clonada como fonte de verdade. 

7. O código deve ser resiliente a mudanças parciais do HTML:

   * evitar quebra total por ausência de seletor
   * usar validações defensivas
   * registrar falhas de parsing quando necessário

8. A persistência deve ser idempotente.
   Se não houver ID externo confiável, gerar um hash estável com base em campos como:

   * nome
   * horário
   * plataforma
   * meeting_id

O guia do projeto sugere usar uma chave/hash estável para `updateOrCreate`.
